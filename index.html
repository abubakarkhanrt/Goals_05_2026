Problem

The problem we solved

Academic transcript verification — high-stakes, repetitive, error-prone when done manually.

👤 What humans need

  • Credits & GPA correct?
  • Layout tampered?
  • Issue/print dates in the future?
  • Risk decision + explanation
  • Follow-up Q&A

🤖 What the agent delivers

  • Deterministic Python checks
  • LLM + page images
  • Structured JSON: legitimacy_score, risk_level, flags
  • Date check: no future issue/print dates vs get_current_datetime
  • Session Q&A cache

From manual review to structured, repeatable assessment

Design

Design philosophy: hybrid agent

AI provides judgment. Python provides proof.

LayerRole
🔄 OrchestratorRoute input, run pipeline, cache session
🔧 ToolsExtract & verify facts from PDF + reference clock
👁 VisionPage images for LLM inspection
🧠 AssessorRisk JSON + plain-language Q&A
🛡 FallbackRule-based scorer — always a result

Layer stack — orchestration on top, proof at the core

Architecture

Agent architecture (ADK Workflow)

Three routes from one gate — verify expands into a linear tool + assess pipeline

Dual path

Dual-path pipeline

Tools read the PDF file; the LLM reads PNG renders — paths never cross incorrectly.

📄 PDF path (tools)

  • Tools read the .pdf file directly
  • Structured JSON: credits, GPA, alignment, dates
  • verify_transcript_dates — no future dates
  • Authoritative for numeric checks

🖼 PNG path (LLM only)

  • Pages rendered to .session_cache/
  • LLM sees images + tool JSON together
  • Tools never read the PNG cache
Demo tip: scanned PDF — tools may fail but vision still helps; tool JSON stays authoritative.

Two paths, one assessment — paths never cross incorrectly

Tool integration

Pattern 1 — Deterministic tools

Workflow nodes call tools directly — not LLM tool-calling — for predictability and testability.

ToolWhat it checks
get_current_datetimeReference UTC clock
verify_transcript_datesNo future dates on PDF
verify_transcript_mathCredits, GPA, logical flags
verify_transcript_spatialAlignment, font consistency
Contract: every tool returns success, flags, and check-specific fields as JSON.

Authoritative facts — LLM cannot override tool flags

Tool integration

Pattern 2 — Workflow nodes as wrappers

Each node runs one tool, saves partial state, and passes results forward in the ADK event chain.

1

Call the pure Python tool

verify_transcript_math(pdf_path) — no ADK dependency inside the tool.

then
2

Persist to session cache

runtime.save_partial(...) so Q&A can reuse results without re-running tools.

then
3

Return Event with state

Event(state={"math_result": result}) — next node receives it automatically.

Same wrapper pattern for spatial and dates — test tools without spinning up ADK.

Thin ADK wrappers around fat, testable tool functions

Output schema

Structured output (Pydantic schema)

Rules scorer and LLM both produce the same shape — downstream systems route on risk_level.

class TranscriptAssessment(BaseModel):
    legitimacy_score: float      # 0.0 – 1.0
    risk_level: Literal[
        "Auto-Approve", "Manual Review", "High Risk"]
    explanation_summary: str | None
    flags: list[str]

LLM backends

LLM layer: local vs cloud

Same workflow graph — swap Ollama or Gemini at the ADK entry point.

🏠 adk run code_local

  • Ollama via LiteLLM
  • Default: gemma4:26b
  • No cloud API key required
  • Privacy-sensitive docs

☁️ adk run code_cloud

  • Gemini via LiteLLM
  • Requires GOOGLE_API_KEY
  • Same workflow, different backend
  • llm_client.py dispatches by env

Both paths: rules fallback if LLM is down, times out, or returns bad JSON.

One graph — swap backend at entry point

Routing

Intent routing & session Q&A

intent.py classifies turns so follow-ups don’t always require a PDF path in the message.

User saysRoute

Extension pattern 1

Add a new verification tool

Plug a deterministic check into the existing PDF pipeline — same contract as math, spatial, and dates.

1

Implement the tool function

verify_pdf_metadata(pdf_path) in a new module — PDF in, JSON out.

then
2

Standardize the return shape

success, flags, findings — scorer and LLM already know this format.

then
3

Wire a workflow node

_run_metadata_check after _run_dates_check in the linear tool chain.

then
4

Feed evidence downstream

Pass result into scoring.py and the LLM evidence payload as metadata_verification.

then
5

Lock it with pytest

Unit tests in tests/test_*.py — no ADK runtime required.

Example: PDF metadata checks (Creator/Producer, modification dates).

One new node — same pattern as dates, math, and spatial

Extension pattern 2

Add a slash command

1

then
2

then
3

then
4

Return local route

Extension pattern 3

Add a workflow route

ADK routes by string keys on Event(route=…) — name every path so flows stay explicit and testable.

1

Define a route constant

In llm_config.py: ROUTE_BATCH = "batch" — one source of truth.

then
2

Return it from the input gate

Event(route=ROUTE_BATCH) when intent matches — e.g. “verify these 5 PDFs.”

then
3

Register the handler node

Add edge in Workflow(edges=[...]): gate → _batch_verify_handler.

then
4

One terminal per outcome

Each route ends at assess, rules, Q&A, or a dedicated sink — no shared “do everything” nodes.

Design rule: if you can’t draw the route on a whiteboard in 30 seconds, split it into smaller routes.

Fan-out from one gate — each route is an explicit string key

Extension pattern 4

Swap or add LLM backend

One workflow graph — pick Ollama or Gemini at the ADK entry point via environment, not code forks.

1

Central dispatch in llm_client.py

assess_transcript() and answer_session_question() route by backend flag.

then
2

Shared prompts in llm_shared.py

Same JSON schema, image resize, and evidence payload — backends differ only by model/API.

then
3

Separate ADK entry points

code_local → TRANSCRIPT_LLM_BACKEND=local · code_cloud → cloud.

then
4

Rules fallback unchanged

Timeout, bad JSON, or API down → same TranscriptAssessment from scoring.py.

Swap backends: adk web → select code_local or code_cloud — no workflow edits.

Backend is a plug — workflow and tools stay identical

Summary

Takeaways for your team

Patterns you can reuse on the next internal agent

🔄

Explicit Workflow, not a chat loop

ADK routes and nodes are testable — you always know what runs next.

🔧

Python proves · LLM explains · Clock validates dates

Tools return JSON facts; the model synthesizes risk and answers questions.

🚦

Route early at the gate

Split verify, Q&A, and local commands before PDF render or LLM calls.

🛡

Always have a fallback

Rules scorer returns the same schema when Ollama/Gemini fails or times out.

🧩

Extend modularly

New tool → workflow node → evidence payload → pytest. Repeat.

📊

Operate what you ship

SQLite audit trail + Grafana on the same service — debuggable in production.

Ship agents that are orchestrated, provable, and observable
