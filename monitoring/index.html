The big picture — data flow

Nothing pushes to Grafana — the agent exposes metrics; Prometheus pulls; Grafana reads Prometheus

Instrumentation code — the helpers

All logic lives in code/observability/metrics.py. Workflow code calls small wrappers:

HelperUsed for
track_task_async()Async nodes (gate, LLM, Q&A)
track_task()Sync nodes (tools, render)
track_cache_io()PDF → PNG file I/O
track_db()SQLite in db/store.py
record_route()verify / qa / local at gate
record_llm_fallback()Rules scorer used instead of LLM
async with track_task_async("llm_assess"):
    assessment = await assess_transcript(...)

with track_task("verify_transcript_math"):
    result = verify_transcript_math(pdf)

with track_db("insert_verification"):
    conn.execute(...)

Pattern

Context manager times the block, records success or error, then exposes via /metrics. If AGENT_METRICS_ENABLED=false, wrappers are no-ops — zero overhead.

Prometheus metric types (and ours)

Counter — only goes up

agent_task_total

agent_route_total

agent_task_errors_total

agent_llm_fallback_total

Grafana uses rate() or shows lifetime total in stat panels.

Gauge — current value

agent_process_cpu_percent

agent_process_memory_bytes

agent_host_cpu_percent

agent_host_memory_*

Updated every 5s by background thread — not on each request.

Histogram — latency buckets

agent_task_duration_seconds

agent_cache_io_seconds

agent_db_seconds

Grafana computes avg, p95 from buckets. Each verify adds samples.

Prometheus configuration

File: monitoring/prometheus/prometheus.yml

scrape_interval: 15s

scrape_configs:
  - job_name: transcript-agent
    metrics_path: /metrics
    static_configs:
      - targets: ["host.docker.internal:9464"]

Check target health: localhost:9090/targets → should show UP.

If DOWN: agent not running, metrics disabled, or wrong port.

Alerts in prometheus/alerts.yml — metrics down 2m, error rate high 5m. View at :9090/alerts

Why CPU shows 0% sometimes

  • Windows cannot measure CPU from a single HTTP scrape
  • We run a daemon thread that samples psutil every 5 seconds
  • Wait ~10s after starting the agent before expecting CPU panels to move
  • 0% process CPU when idle is normal — Python waits for requests
  • Host CPU may stay >0 if Ollama or Docker are busy
